# 구체화하기: 배민배달 배달 도메인과 이벤트 데이터 모델

> 우아한형제들 기술블로그 글 [*우리 팀은 카프카를 어떻게 사용하고 있을까*](https://techblog.woowahan.com/17386/)
> (2024-05-30, 김나은)에서 다룬 주문–배달 도메인을 **충실히 재구성**한 문서로, Transactional Outbox + CDC
> 발행 경로는 압축된 보조 섹션으로 정리한다.
>
> 이 문서는 영문 원본 [`woowahan-delivery-service.md`](woowahan-delivery-service.md)의 한국어 번역본이다.
> 구체화를 위해 재구성한(원문에 명시되지 않은) 부분은 **[추정]**으로 표시한다.

## 1. 맥락과 범위

**딜리버리서비스팀**은 **배민배달**(배달의민족의 자체 배달)을 운영하며 **하루 100만 건 이상의 배달을
중계**한다. 이 팀의 핵심 특징이자 데이터 모델 전체를 이해하는 열쇠는 주문의 발신자가 아니라
**중계자**라는 점이다:

> 배달의 민족에서 제공하는 여러 주문서비스(배민배달, B마트, 배민스토어)의 배민배달을 받아 여러 배달서비스
> 중 하나로 분배하고, 배달과정을 중계하고 관리하는 역할을 합니다.

이 팀은 여러 상위 주문 서비스로부터 주문을 **받아**, 각 주문을 여러 하위 배달 서비스 중 하나에 **분배**하고,
배달 과정을 **중계·관리**한다. 범위에 포함되는 서버군은 두 개이며 각각 N개의 인스턴스로 구성된다:
**주문/배달서버**는 상태를 MySQL에 저장하고 이벤트를 발행하며, **분석서버**는 이를 소비한다(범위 외).

이 문서는 **도메인과 그 데이터**에 집중한다: 배달이 *무엇*인지, 상위 주문과 어떻게 관계 맺는지, 그
라이프사이클이 어떻게 이벤트로 모델링되는지. 신뢰성 있는 발행 메커니즘(Transactional Outbox + Debezium
CDC, 블로그 [1]절)은 §5에 압축된 형태로만 등장한다.

## 2. 도메인 모델

### 2.1 액터와 시스템

| 액터 / 시스템 | 역할 | 팀과의 관계 |
|---|---|---|
| **상위 주문 서비스** — 배민배달 · B마트 · 배민스토어 | 주문을 발생시킴 | 팀은 이들의 주문 이벤트를 **받음**; 주문을 **소유하지 않음** |
| **딜리버리서비스팀** (이 시스템) | 주문마다 **배달**을 생성·관리; 분배·중계 | 배달을 **소유함** |
| **하위 배달 서비스** | 실제로 배달을 수행; 라이더를 공급 | 배달마다 **분배(distribution)**로 하나가 선택됨 |
| **라이더** | 픽업하고 배달함 | **배차(dispatch)** 시 배정됨 *[추정 — §3 참고]* |
| **운영자 / CS** | 분배 규칙 조정, 모니터링 | 런타임에 분배규칙(distribution rules)을 변경 |

### 2.2 주문 vs 배달 (핵심 구분)

팀이 **중계자**이기 때문에, 식별자·소유자·라이프사이클이 서로 다른 두 가지를 절대 혼동해서는 안 된다:

| | **주문** | **배달** |
|---|---|---|
| 식별자 | **주문식별자 `orderId`** | **배달식별자 `deliveryId`** |
| 소유 주체 | 상위 주문 서비스 | **이 팀** |
| 라이프사이클 | 생성 / 결제 / 취소 (상위) | 생성 → 배차 → 픽업 → 완료 |
| 보관 위치 | 팀 데이터에 참조로 저장됨 | 팀의 **애그리거트 루트** (MySQL) |

팀은 **승인된 주문 1건을 처리하기 위해 배달 1건을 생성**한다 *(주문 1 → 배달 1; 여기서는 1:1로 취급 —
[추정]; 블로그는 분할 배달을 다루지 않음)*. 이 구분이 §6의 토픽·키 구성을 좌우한다.

### 2.3 분배와 배차

배달의 *수행 주체*를 결정하는 두 도메인 단계가 있으며, 각각 애그리거트의 한 필드를 채운다:

- **분배(Distribution)** — 배달 서버가 운영자가 런타임에 바꿀 수 있는 인메모리 **분배규칙(distribution
  rules)**을 사용해 배달에 대해 **배달 서비스 하나**를 선택한다. → `deliveryServiceId`를 설정.
  *(규칙 변경을 서버 간에 전파하는 것은 블로그의 Event-Bus 서브시스템으로 범위 외이며, 그 결과만 데이터
  모델에 반영된다.)*
- **배차(Dispatch)** — 선택된 배달 서비스의 **라이더**가 배정되며 **배차완료(DispatchCompleted)** 이벤트가
  발생한다. → `riderId`를 설정. *[추정 — 블로그는 배차완료를 이벤트로 언급하지만 라이더 필드는 명시하지
  않음.]*

### 2.4 배달 라이프사이클

배달은 **순서가 있는 상태**를 거치며 각 단계마다 이벤트를 발행한다
("배달은 생성, 배차, 픽업, 완료 등 순서를 가지고 진행되며, 특정 행위마다 배달이벤트를 발행"). 이후의 모든
설계 결정을 좌우하는 도메인상 주의할 점이 두 가지 있다:

- **거의 동시에 발생하는 이벤트.** 배차완료와 픽업준비요청은 거의 동시에 발생할 수 있다. 프로듀서는
  배차완료를 *먼저* 발행하고 픽업준비요청을 발행하지만, 컨슈머는 역순으로 수신해 어느 것이 실제로 먼저
  일어났는지 놓칠 수 있다. → **배달별로 순서를 보장해야 한다**(문제 P1, §4).
- **취소는 언제든 도착할 수 있다.** 상위의 주문취소는 배달취소를 발생시키며, 배달은 그 시점의 비종료
  상태에서 `CANCELLED`로 전이한다. 이 이벤트가 누락되면 취소된 배달이 계속 진행된다. → **이벤트는
  누락되어선 안 된다**(문제 P2, §4).

### 2.5 도메인 불변식

1. **하나의 배달, 하나의 진실 소유자.** MySQL의 `Delivery` 애그리거트가 배달 상태의 진실 원천이며,
   이벤트는 상태 변화에 *대한 사실*일 뿐 상태 그 자체가 아니다.
2. **취소를 제외하면 상태는 단조 증가.** `CREATED → … → DELIVERED`로 전진만 하며, `CANCELLED`가 유일한
   "옆길" 전이로 비종료 상태에서만 가능하다.
3. **배달별 순서는 반드시 보존; 배달 간 순서는 무관.** *같은* `deliveryId`의 두 이벤트는 의미 있는 순서를
   가지지만, *다른* 배달의 두 이벤트는 그렇지 않다.
4. **주문은 참조될 뿐 소유되지 않음.** 팀은 `orderId`와 어느 상위 서비스가 보냈는지를 저장하지만, 주문
   자체의 라이프사이클은 상위에 속한다.

## 3. 데이터 모델 (`Delivery` 애그리거트 중심)

### 3.1 `Delivery` 애그리거트

애그리거트 루트. 블로그가 암시하거나 직접 언급한 필드는 표시가 없고, 모델을 구체화하기 위해 재구성한
필드는 **[추정]**으로 표시한다.

```
Delivery                      -- 애그리거트 루트; MySQL의 배달 1건당 1행
  deliveryId        : string  -- 배달식별자 · PK · 이벤트 키 & 파티션 키
  orderId           : string  -- 주문식별자 · 상위 참조(FK 유사, 소유 아님)
  orderServiceType  : enum    -- BAEMIN_DELIVERY | B_MART | BAEMIN_STORE   (어느 상위가 보냈는지)
  deliveryServiceId : string  -- 분배(distribution)로 선택됨
  riderId           : string? -- 배차완료 시 설정                          [추정]
  status            : enum    -- CREATED | ASSIGNED | PICKUP_REQUESTED | PICKED_UP | DELIVERED | CANCELLED
  pickup            : Place   -- 음식점 위치                               [추정, 최소]
  dropoff           : Place   -- 고객 위치                                 [추정, 최소]
  createdAt         : ts
  assignedAt        : ts?     -- 배차완료 시 기록
  pickupRequestedAt : ts?     -- 픽업준비요청 시 기록
  pickedUpAt        : ts?     -- 픽업 시 기록
  deliveredAt       : ts?     -- 완료 시 기록
  cancelledAt       : ts?     -- 배달취소 시 기록
```

각 라이프사이클 이벤트는 **하나의 상태 전이 + 그것이 설정하는 타임스탬프/필드**에 대응한다 — 즉 이벤트는
*이 애그리거트의 델타(delta)*이다(§3.4).

### 3.2 관계

```
   Order (상위, 참조됨)                            DeliveryService (분배 대상)
     orderId  ◀───────────────┐                   deliveryServiceId
     orderServiceType          │ 참조                    ▲
                               │ (N:1, 실질 1:1)        │ 분배 (N:1)
                        ┌──────┴────────────────────────┴──────┐
                        │                Delivery               │
                        │  deliveryId (PK)                      │
                        │  orderId  ───────────► Order          │
                        │  deliveryServiceId ──► DeliveryService│
                        │  riderId  ───────────► Rider  [추정]  │
                        │  status, timestamps…                  │
                        └───────────────────┬───────────────────┘
                                            │ 배차 시 배정 (N:1)
                                            ▼
                                        Rider  [추정]
                                          riderId
```

- **주문 → 배달: 1:1** (배달 1건이 승인된 주문 1건을 처리) *[엄격한 1:1은 추정]*.
- **배달 → 배달 서비스: N:1** (여러 배달이 한 서비스로 분배됨).
- **배달 → 라이더: N:1** *[추정]* (한 라이더가 시간에 걸쳐 여러 배달을 수행; 배정 시점엔 하나).

### 3.3 이벤트 키: 주문식별자 vs 배달식별자

블로그는 순서가 필요한 식별자를 키로 삼아 순서를 보장한다고 명시한다
("주문식별자, 배달식별자 등과 같이 순서관리가 필요한 식별자를 키로 관리하여 순서를 보장"). 이
서브시스템에서 그 규칙은 §2.2의 소유 경계를 따른다:

- **`orderId`(주문식별자)**는 상위 **`order`** 토픽의 키이며, 주문별 순서를 보장한다.
- **`deliveryId`(배달식별자)**는 **`delivery`** 토픽의 키이자 *모든* 라이프사이클 이벤트의 파티션 키이며,
  배달별 순서를 보장한다. 토픽 구성은 §6 참고.

애그리거트는 둘 다 가지지만, 자기 식별성과 이벤트 키는 모두 `deliveryId`이다.

### 3.4 애그리거트의 델타로서의 라이프사이클 이벤트

각 상태 변경 이벤트 = 전이 + 그것이 기록하는 필드. 카탈로그(한국어 용어는 블로그 원문):

| 이벤트 | 한국어 | 전이 | 기록/설정하는 값 |
|---|---|---|---|
| `DeliveryCreated` | 배달생성 | → `CREATED` | deliveryId, orderId, orderServiceType, pickup, dropoff, createdAt |
| `DispatchCompleted` | 배차완료 | `CREATED`/`ASSIGNED` → `ASSIGNED` | deliveryServiceId, riderId, assignedAt |
| `PickupRequested` | 픽업준비요청 | → `PICKUP_REQUESTED` | pickupRequestedAt |
| `PickedUp` | 픽업 | → `PICKED_UP` | pickedUpAt |
| `Delivered` | 완료 | → `DELIVERED` | deliveredAt |
| `DeliveryCancelled` | 배달취소 | 비종료 → `CANCELLED` | cancelledAt, reason |

> **명명 메모.** 픽업준비요청은 문자 그대로 "준비(prep)"의 뉘앙스를 담지만, 이 문서는 더 짧은
> `PickupRequested`를 영문 핸들로 사용한다.

**이벤트 엔벨로프** *(재구성 — 블로그는 스키마를 제공하지 않음; 직렬화는 Avro + Schema Registry, [추정]):*

```
eventId       : UUID       # 멱등성 핸들
eventType     : string     # "DispatchCompleted" | ...
aggregateType : "delivery" # `delivery` 토픽으로 라우팅
aggregateId   : string     # = deliveryId → Kafka 메시지 키
occurredAt    : timestamp
version       : int        # 추가 전용(additive-only) 스키마 진화
payload       : <이벤트별 레코드, 위의 델타들>
```

**대표 페이로드** (나머지도 같은 형태를 따름 — 델타 + 컨슈머 조인을 위한 키):

```jsonc
// DispatchCompleted (배차완료)
{
  "deliveryId": "DLV-20260628-000123",
  "orderId": "ORD-20260628-998877",
  "deliveryServiceId": "DS-ALPHA",
  "riderId": "RDR-4412",                 // [추정]
  "assignedAt": "2026-06-28T11:02:13.412+09:00"
}

// DeliveryCancelled (배달취소)
{
  "deliveryId": "DLV-20260628-000123",
  "orderId": "ORD-20260628-998877",
  "reason": "ORDER_CANCELLED",           // 상위 주문취소 → 배달취소
  "cancelledAt": "2026-06-28T11:05:51.007+09:00"
}
```

### 3.5 상태 머신

```
  배달생성        배차완료           픽업준비요청          픽업           완료
 ─────────▶ CREATED ─────▶ ASSIGNED ─────▶ PICKUP_REQUESTED ─────▶ PICKED_UP ─────▶ DELIVERED (종료)
                │                                                                   ▲
                └──────────── (전진만; 거의 동시 발생 배차완료 ↔ 픽업준비요청) ┘

  비종료 상태 ──── 배달취소 (DeliveryCancelled) ────▶ CANCELLED (종료)
```

컨슈머는 이 머신에 대해 전이를 검증하여, 현재 상태에 맞지 않는 이벤트를 보류/거부한다 — P1에서 설명한
재정렬에 대한 방어이다.

## 4. 두 가지 문제 (발행 경로가 존재하는 이유)

위의 데이터 모델은 이벤트가 컨슈머에 **순서대로** 그리고 **누락 없이** 도달할 때에만 안전하다.

- **P1 · 순서 보장.** 거의 동시에 발생하는 배차완료/픽업준비요청이 역순으로 소비되면 배달별 상태
  로직이 깨진다. → **`deliveryId`별** 순서를 보장.
- **P2 · 무손실과 정합성.** MySQL 상태 변경과 Kafka 이벤트는 원자적이어야 한다; 배달취소가
  누락되면 취소된 배달이 계속 진행된다. → 이중 쓰기 없음, 무손실 재시도.

이 두 문제가 §5의 압축된 메커니즘을 필요하게 하며, 그 메커니즘으로 해결된다.

## 5. 데이터 모델의 발행 (압축)

이 절은 §3의 이벤트를 운반하는 Transactional Outbox + Debezium CDC 경로를 요약한다. 팀의 서술은 원문을
참고한다.

- **아웃박스 브리지.** 각 상태 변경은 비즈니스 행과 아웃박스 행을 **하나의 로컬 MySQL 트랜잭션**으로
  기록한다(이중 쓰기 없음). 아웃박스 테이블은 **삽입 전용(insert-only)**이며 저장된 순서를 보존한다.

  ```sql
  CREATE TABLE delivery_outbox_1 (        -- _1.._N 샤드
    id BIGINT AUTO_INCREMENT PRIMARY KEY, -- 단조 증가 = 커밋/binlog 순서
    aggregate_type VARCHAR(64),           -- 'delivery' → 라우팅 토픽
    aggregate_id   VARCHAR(64),           -- = deliveryId → Kafka 키
    event_type     VARCHAR(64),
    event_id       CHAR(36) UNIQUE,       -- → 컨슈머 멱등성
    payload        JSON,
    occurred_at    DATETIME(3)
  );                                       -- INSERT 전용; UPDATE/DELETE 없음
  ```

- **처리량을 위한 샤딩.** 테이블은 식별자 기준으로 `delivery-outbox1/2/3`처럼 분리된다. 안정적인
  `shardIndex = (hash(deliveryId) % N) + 1`를 통해 **같은 `deliveryId` → 같은 샤드 테이블**이 된다
  *[정확한 함수는 추정]*. 이들은 **토픽이 아니라 테이블**이며, 모든 샤드가 단일 `delivery` 토픽으로 흘러든다.
- **CDC 발행, 샤드당 태스크 1개.** **Debezium MySQL source connector**가 샤드 테이블별로 커밋된
  **binlog**를 테일링(로그 테일링)하며, **`tasks.max=1`**(블로그가 명시한 순서 보장 장치)로 동작하고,
  `aggregate_type`으로 라우팅하고 `aggregate_id`로 키를 정한다. 발행은 저장된 오프셋부터 재시도하는
  **최소 1회(at-least-once)** 방식이다.
- **순서 체인(P1).** `같은 deliveryId → 같은 샤드 테이블 → 단일 태스크 커넥터 → 같은 파티션(key=deliveryId)
  → 같은 컨슈머` ⇒ 배차완료가 항상 픽업준비요청보다 앞선다. 이 사슬 중 하나라도 끊으면(태스크 여러 개 /
  불안정한 샤드 / 잘못된 키) 재정렬 문제가 다시 발생한다.
- **정합성(P2), 그리고 정정.** 블로그는 "메시지 발행에 실패하면 아웃박스테이블의 데이터도 롤백"이라고
  하는데 — **기술적으로 엄밀하지 않다.** 로그 테일링 CDC 설계에서는 발행 시점에 롤백되는 것이 없다: 아웃박스
  행은 *먼저 비즈니스 쓰기와 원자적으로 커밋*되고, 그 다음 Debezium이 *이미 커밋된* binlog에서 최소 1회
  재시도로 발행한다. 정합성은 분산 트랜잭션이 아니라 **로컬 원자성 + 무손실 재시도**에서 온다.
  *(이 지점에서 충실함이란 원문의 부정확함을 짚어내는 것이다.)*

## 6. 토픽 / 파티션 / 키 구성

| 토픽 | 키 | 운반 내용 | 비고 |
|---|---|---|---|
| `order` | `orderId` (주문식별자) | 상위 주문 이벤트 | 참조될 뿐 소유 아님 |
| `delivery` | **`deliveryId` (배달식별자)** | §3.4의 모든 라이프사이클 이벤트 | `P`개 파티션, 컨슈머 병렬성에 맞춰 사이징 |
| `analysis` | `deliveryId` | 재발행·전처리됨 | 범위 외 |

N개의 샤드 커넥터가 `delivery`에 기록한다. 이와 별개로 `partition = hash(deliveryId) % P`가 한 배달의 모든
이벤트를 한 파티션에 떨어뜨린다. 프로듀서 샤딩과 브로커 파티셔닝은 **같은** 키에 대한 독립적인 해시
함수이므로, 서로 조율하지 않아도 같은 키의 이벤트가 같은 파티션에 모인다. *[추정 — 파티션 수는 제시되지 않음. `N`(샤드
테이블)과 `P`(파티션)는 독립적인 단계를 튜닝한다: 프로듀서/CDC 처리량 vs. 컨슈머 병렬성. 배달별 순서는 임의의
`P ≥ 1`에서 성립한다; `P`는 컨슈머 병렬성에, `N`은 CDC 처리량에 맞춰 사이징한다.]*

## 7. 누락된 정보와 추정 (정직성 기록)

이 재구성에서 단순히 빈칸을 채우는 것이 아니라 **원문의 서술과 어긋나는** 유일한 지점은 아웃박스 롤백
메커니즘이다.

블로그는 *"메시지 발행에 실패하면 아웃박스테이블의 데이터도 롤백되기 때문에"*라고 적는데, 이는 Kafka
발행이 실패하면 아웃박스 행이 롤백된다는 뜻으로 읽힌다. 그러나 이는 **같은 글이 설명하는 메커니즘과
모순된다**: Debezium **로그 테일링**은 로컬 트랜잭션이 커밋된 *이후* **이미 커밋된** binlog를 읽으므로,
발행을 감싸 롤백할 트랜잭션 자체가 없다. 정확한 모델은 **커밋 후 테일링(commit-then-tail) + 최소
1회(at-least-once) 재시도**이다(§5). 확신도는 **높다**. 이 모순은 글 내부에서 발생한다. binlog CDC에 대한
설명은 명확하고, 요약 문장 하나만 엄밀하지 않다. 여기서 충실한 재구성이란 그럴듯한 서술보다
메커니즘 편에 서는 것이며, 오류를 조용히 반복하기보다 드러내어 말하는 것이다.
